# 多智能体编排增强设计

## 目标

把 TradingAgents 和 CrewAI 中适合当前系统的多智能体管理机制吸收到 `D:\swap\orch`，让任务规划更聪明、更准确、更一致，并减少无用执行和无意义等待。

本次设计只面向现有 Node.js 编排系统，不引入 LangGraph、CrewAI 或新的第三方依赖，不改运行入口，不重写大文件。

## 外部系统结论

### TradingAgents 可借鉴机制

TradingAgents 的核心不是让所有 Agent 自由聊天，而是把复杂交易任务固定拆成多类专家节点：

- 分析师先产出不同维度的材料。
- 正方和反方研究员进行有界辩论。
- 风险团队从激进、中性、保守视角复核。
- 组合经理基于前面材料做最终裁决。
- 决策日志、checkpoint 和历史经验会被保存，供后续任务复用。

可集成到 orch 的部分：

- 有界辩论，而不是无限会议。
- 风险复核和最终经理裁决。
- 结构化决策日志。
- 复杂任务必须留下可复核的编排理由。

不集成的部分：

- 不引入 LangGraph。
- 不照搬金融交易领域的固定角色名。
- 不新增 Python 运行链路。

参考来源：

- https://github.com/TauricResearch/TradingAgents

### CrewAI 可借鉴机制

CrewAI 的核心是把协作模式分成两层：

- `Crew` 负责角色、任务、上下文、工具、记忆和执行顺序。
- `Flow` 负责事件、状态、分支、条件和精确控制。

CrewAI 还内置了很多硬校验：

- 层级流程必须有 manager。
- 顺序流程的任务必须绑定 agent。
- 任务上下文不能引用未来任务。
- 异步任务和条件任务有明确约束。
- 层级流程由 manager 做分派、验证和推进。

可集成到 orch 的部分：

- `process` 流程类型。
- 层级 manager 调度和复核。
- 本地计划校验器。
- 条件、依赖、上下文合法性检查。
- 任务契约和质量门。

不集成的部分：

- 不把 CrewAI 当运行时依赖。
- 不引入 YAML Crew 配置体系。
- 不改当前 SQLite 任务模型。

参考来源：

- https://github.com/crewAIInc/crewAI
- https://docs.crewai.com/en/concepts/processes
- https://docs.crewai.com/en/concepts/flows

## 当前 orch 基础

当前系统已经具备以下能力：

- `planner.js` 能识别简单、标准、复杂和待确认任务。
- `planner.js` 能生成复杂任务保底多员工计划，并在必要时插入会议。
- `runner.js` 已支持会议发言、会议总结、知识检索、任务目录上下文注入。
- `engine.js` 已支持依赖驱动、并发槽、命名锁、质量门循环、重规划和实时日志。
- `api.js` 和前端画布已能展示员工、耗时、日志预览和节点详情。
- `server.js` 已支持用户选择 A/B/C 后继续规划。

主要短板不是执行引擎，而是规划层缺少更强的流程模型和硬复核规则：

- 复杂任务是否需要会议的判断仍偏粗。
- 会议结构还没有固定为辩论、风险复核、裁决。
- 计划生成后缺少类似 CrewAI 的本地合法性校验。
- 复杂任务的编排理由没有结构化沉淀。
- 速度慢主要来自规划期调用和不必要会议，而不是依赖执行本身。

## 推荐设计

采用“机制集成”方案：保留现有 orch 架构，把 TradingAgents 和 CrewAI 的好机制转成轻量本地规则。

### 1. 增加流程类型

在计划结果中增加轻量元数据：

```js
plan.process = {
  type: 'fast' | 'sequential' | 'hierarchical' | 'debate' | 'risk_review' | 'ask_user',
  reason: '选择该流程的中文理由',
  manager_role: '可选的经理角色 id',
  debate_rounds: 0 | 1,
  risk_review: true | false
}
```

流程选择规则：

- `fast`：高置信简单任务，直接少步骤执行，不开会。
- `sequential`：标准任务，按依赖顺序推进，不开复杂会议。
- `hierarchical`：复杂但风险不高的任务，由产品、架构或项目管理角色做经理裁决。
- `debate`：复杂且存在方案分歧的任务，进入有界辩论。
- `risk_review`：金融、交易、权限、安全、数据风险类任务，必须有风险或安全复核。
- `ask_user`：需求过宽或歧义明显，先给用户 A/B/C 选择。

这些元数据只进入计划 JSON 和事件日志，不要求改数据库结构。

### 2. 增加本地计划复核器

新增 `validateCrewPlan(plan, roles, intent)`，放在 `planner.js` 附近，保持小函数实现。

复核规则：

- 每个可执行步骤必须绑定真实存在的员工角色。
- `deps` 不能引用不存在、未来、自身或重复步骤。
- 标准顺序计划不能出现无员工步骤。
- 复杂计划必须有经理裁决或会议裁决节点。
- 复杂计划必须有验收、测试或评审节点。
- 股票、交易、金融、支付、权限、安全类任务必须包含安全、风险或测试角色。
- 条件步骤不能作为第一个根步骤。
- 带循环质量门的步骤必须有明确 `expected_outcome`。
- 多个并发步骤明显修改同一类文件时，应提示添加 `lock`，但初版只记录警告，不阻断。

复核结果结构：

```js
{
  ok: true,
  errors: [],
  warnings: [],
  repaired: false
}
```

处理策略：

- `errors` 存在时，优先本地修复。
- 本地无法修复时，复杂任务走复杂保底计划。
- 不新增一次 LLM 复核，避免变慢。

### 3. 升级会议为有界裁决流程

当前会议不再只是多人发言，应升级为固定议程：

1. 主持人明确任务目标和成功标准。
2. 产品或经理角色说明用户价值和范围边界。
3. 架构或后端角色说明系统方案。
4. 前端或体验角色说明用户界面和交互。
5. 安全、风险或测试角色提出反方质询。
6. 主持人输出裁决、行动项、验收口径和风险清单。

触发规则：

- 标准任务默认不开会。
- 复杂任务才开会。
- 高风险复杂任务必须开会。
- 如果用户已选择 A/B/C 且范围明确，只开轻量裁决，不重复大讨论。

轮数限制：

- 默认一轮初评加一轮裁决。
- 不做多轮自由讨论。
- 不因会议结果反复调用规划模型。

### 4. 增加编排决策日志

借鉴 TradingAgents 的 decision log，但使用当前事件系统和任务目录文件。

新增事件类型：

```js
{
  type: 'orchestration_decision',
  process_type: 'risk_review',
  reason: '为什么选择该流程',
  attendees: ['product-manager', 'backend-architect', 'security-reviewer'],
  required_roles: ['backend', 'frontend', 'security', 'testing'],
  rejected_roles: ['不选择的角色及原因'],
  validation_errors: [],
  validation_warnings: []
}
```

同时在 `task_plan.md` 或 `方案.md` 增加“编排决策”段落：

- 任务复杂度判断。
- 选择的流程类型。
- 参会员工。
- 关键依赖。
- 风险复核依据。
- 未采用角色和原因。

这样用户可以复核“为什么这样编排”，而不是只看到节点结果。

### 5. 速度优化策略

规划速度优化优先做本地判断，避免增加模型调用：

- 简单高置信任务保持零规划模型或一次极短规划。
- 标准任务最多一次规划模型。
- 复杂任务最多一次规划模型加本地复核。
- 只有计划不合法时走本地复杂保底，不做第二次模型返工。
- 会议只在复杂或高风险任务触发。
- 会议参会角色限制在 4 到 5 个。
- 辩论轮数默认 1。
- 校验和修复全部本地完成。

### 6. 前端画布展示

画布节点继续使用现有实时日志能力，只补充流程信息：

- 节点标题保持中文短标题。
- 节点显示“员工：部门 / 姓名 / 角色”。
- 任务结束显示执行耗时。
- 节点详情展示实时日志、编排理由、上游依赖、预计交付物。
- 会议或裁决节点显示 `process.type` 和参会员工。

本次设计不要求重做画布，只补足计划元数据和展示字段。

## 数据流

1. 用户创建任务。
2. `planner.js` 先执行本地意图判断。
3. 若歧义明显，返回 `ask_user` 和 A/B/C。
4. 若需求明确，生成计划。
5. `validateCrewPlan()` 做本地复核。
6. 复核不通过时本地修复或走复杂保底。
7. 写入 `orchestration_decision` 事件。
8. 复杂任务进入有界会议。
9. `runner.js` 执行会议裁决或任务步骤。
10. `engine.js` 按依赖、锁和质量门执行。
11. 前端画布实时展示角色、日志和耗时。

## 错误处理

- 找不到适合员工时，任务进入 `awaiting_input`，提示用户选择范围或补充要求。
- 计划依赖非法时，本地修复；修复失败则重新生成复杂保底计划。
- 高风险任务缺少安全或测试角色时，强制插入复核步骤。
- 会议失败时，保留已产生的发言日志，并让经理裁决节点输出待确认问题。
- 用户选择 A/B/C 后，清理旧步骤并按选择重新规划。

## 非目标

- 不直接接入 CrewAI。
- 不直接接入 LangGraph。
- 不新增数据库迁移。
- 不把当前任务系统改成 YAML 配置系统。
- 不重写 `engine.js` 执行器。
- 不扩大到 `D:\channl` 或其他项目。

## 成功标准

- 复杂股票交易网站会被识别为复杂高风险任务。
- 系统会自动安排产品、架构或后端、前端、安全或风险、测试等必要角色。
- 复杂任务会触发会议或裁决，而不是单步执行。
- 每个节点能看出由哪个员工执行。
- 计划能解释为什么这么编排。
- 本地复核能阻止明显不合理计划。
- 标准任务不会因为新机制明显变慢。
- 全量测试通过。
